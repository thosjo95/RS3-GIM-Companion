3. SSL & HTTPS

SSL is handled by a Cloudflare Origin Certificate — a certificate issued by Cloudflare to secure the connection between Cloudflare's edge and the origin server (Hetzner). It is not a public CA certificate.

PropertyDetail
Certificate typeCloudflare Origin Certificate (15 years)
Certificate file on server/etc/ssl/cloudflare/origin.pem
Private key file on server/etc/ssl/cloudflare/origin.key
Local backup locationC:\Users\Thomas\OneDrive\RS3 GIM Companion\
Cloudflare SSL modeFull (Strict) — encrypts end-to-end, validates origin cert
Port 80 behaviourNginx redirects all HTTP → HTTPS (301)

Nginx SSL block (summary)

server {
    listen 443 ssl;
    server_name groupiron.com www.groupiron.com;

    ssl_certificate     /etc/ssl/cloudflare/origin.pem;
    ssl_certificate_key /etc/ssl/cloudflare/origin.key;

    # Serve React build
    root /root/RS3-GIM-Companion/client/dist;
    index index.html;
    try_files $uri $uri/ /index.html;

    # Proxy /api to Node.js
    location /api/ {
        proxy_pass http://localhost:3001;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
    }
}

server {
    listen 80;
    server_name groupiron.com www.groupiron.com;
    return 301 https://$host$request_uri;
}
If the cert ever needs renewing: Generate a new Origin Certificate in the Cloudflare dashboard → SSL/TLS → Origin Server → Create Certificate. Save both files and copy them to /etc/ssl/cloudflare/ on the server, then sudo systemctl reload nginx.

6. Deploying Changes

The full workflow is: edit locally → commit → push to GitHub → SSH into server → run deploy script.

Step-by-step

  1. Make your changes in VS Code or with Claude in the Claude desktop app
  2. Claude commits and pushes to GitHub (or do it yourself: git add . && git commit -m "message" && git push)
  3. SSH into the server: ssh root@178.104.126.158
  4. Run the deploy script: bash /root/deploy.sh
  5. The script pulls changes, builds the React frontend, and restarts PM2

What deploy.sh does

cd /root/RS3-GIM-Companion
git pull origin main
npm install --prefix server     # installs any new server packages
npm run build --prefix client   # Vite build → client/dist/
pm2 restart app

Checking the deploy worked

pm2 status          # should show "online"
pm2 logs app --lines 20   # check for startup errors
curl localhost:3001/api/health   # should return {"status":"ok",...}
Typical deploy time: 30–60 seconds. The site is briefly unavailable during PM2 restart (<2 seconds).

8. Requesting Features with Claude

The project is developed using Claude Code (Anthropic's AI coding assistant) inside the Claude desktop app. Claude has access to all project files and can read, edit, and commit code.

How to ask for a feature

  1. Open the RS3 GIM Companion project in Claude desktop app (the working directory is set automatically)
  2. Describe what you want in plain English. Be specific about: where it should appear, what it should show, what data it uses
  3. Claude will read the relevant files, make the changes, and commit + push
  4. You then SSH into the server and run bash /root/deploy.sh

Tips for good feature requests

  • Reference existing things: "Like the Skills tab but for bosses" is clearer than "add a boss tab"
  • Describe the data source: "This should come from hiscores" or "detect it from the activity feed"
  • Describe the layout: "A table with players as columns" or "a grid of cards"
  • One session = one topic: Mixing unrelated tasks in one message works fine, but very large changes across many files are easier to review in separate sessions
  • After deploy, test it: Report back if something looks wrong and Claude will fix it

Example prompts that worked well

  • "Add a Leaderboards tab with a Firsts section showing ~40 predefined milestones as cards. Auto-detect from activity feed. Show who achieved each first, or 'not yet' if nobody has."
  • "Replace the emoji skill icons with the official RS3 wiki images from runescape.wiki/images/{Skill}.png"
  • "The Active This Week stat doesn't work on fresh databases. Fix it so it falls back to the earliest snapshot instead of returning null."

9. Database Reference & SQL

The database is SQLite, located at /root/RS3-GIM-Companion/server/data/gim.db.

Connect to the database

ssh root@178.104.126.158
sqlite3 /root/RS3-GIM-Companion/server/data/gim.db

Exit with .quit. Use .tables to list tables. Use .mode column and .headers on for readable output.

Schema overview

TableKey columnsNotes
groupsid, name, gim_type, gim_size, password_hash, last_activitypassword_hash = NULL means unclaimed
playersid, rsn, group_id, combat_level, last_synced, stats_json, activities_jsonstats_json = hiscores CSV parsed as JSON; activities_json = RuneMetrics array

Useful SQL queries

See all groups and their claim status

SELECT id, name, gim_type, (password_hash IS NOT NULL) as claimed, last_activity
FROM groups;

Get a group's secret (if you forgot it — you can't recover it, but you can reset it)

-- You can't recover the secret (it's hashed). To reset so a group becomes unclaimed:
UPDATE groups SET password_hash = NULL WHERE name = 'True Deciples';

See all players in a group

SELECT p.rsn, p.combat_level, p.last_synced, s.level as total_level, s.xp as total_xp
FROM players p
LEFT JOIN skills s ON s.player_id = p.id AND s.skill_name = 'Overall'
WHERE p.group_id = 1
ORDER BY s.xp DESC;

Check a player's clue scroll counts

-- stats_json stores hiscores including activities (clue counts)
SELECT rsn, stats_json FROM players WHERE rsn = 'YourRSN';
-- Then look in the JSON for "Clue Scrolls All" etc.

List all drops for a group

SELECT p.rsn, d.item_name, d.notes, d.dropped_at
FROM drops d
JOIN players p ON p.id = d.player_id
WHERE p.group_id = 1
ORDER BY d.dropped_at DESC;

Count goals by status

SELECT status, COUNT(*) FROM goals GROUP BY status;

Show XP snapshots for a player (for debugging weekly XP)

SELECT snapshot_date, total_xp
FROM snapshots
WHERE player_id = (SELECT id FROM players WHERE rsn = 'YourRSN')
ORDER BY snapshot_date DESC
LIMIT 10;

Wipe everything and start fresh

DELETE FROM drops; DELETE FROM item_requests; DELETE FROM goals;
DELETE FROM goal_contributors; DELETE FROM snapshots;
DELETE FROM skills; DELETE FROM players; DELETE FROM groups;
-- Then hard-reset autoincrement:
DELETE FROM sqlite_sequence;

Reset a group to unclaimed without wiping data

UPDATE groups SET password_hash = NULL WHERE id = 1;

Back up the database

cp /root/RS3-GIM-Companion/server/data/gim.db ~/gim_backup_$(date +%Y%m%d_%H%M).db

Restore from backup

pm2 stop app
cp ~/gim_backup_20250505_1200.db /root/RS3-GIM-Companion/server/data/gim.db
pm2 start app

Backend (server/)

FileWhat it does
database.jsOpens SQLite file, runs CREATE TABLE IF NOT EXISTS for all tables, runs safe ALTER TABLE migrations for new columns. Exports db singleton with added db.runTransaction() helper.
index.jsExpress app entry. Mounts all routes. Runs a node-cron job at midnight to snapshot all players' XP.
routes/groups.jsCRUD for groups. Key endpoints: GET /:id (returns group + players + skills + weekly XP gains), POST /setup (multi-step group import from hiscores), POST /:id/claim (generates secret, stores hash, returns plain text once), POST /:id/verify (checks password), GET /search (find groups by name).
routes/players.jsCRUD for players. Key endpoint: POST /:id/sync — calls fetchHiscores + fetchRuneMetrics, upserts skills, stores stats_json + activities_json, calls autoLogActivityDrops() to auto-detect drops from activity text.
routes/goals.jsCRUD for goals. GET /suggestions/:groupId calls generateSuggestedGoals() from runescape.js.
routes/drops.jsCRUD for drops and item_requests. Drops are filtered by group_id via player join.
services/runescape.jsAll Jagex API integrations. fetchHiscores(rsn): fetches CSV from hiscores, parses 30 skills + 11 activities (clue scrolls etc.) into structured JSON. fetchRuneMetrics(rsn): fetches recent activities + quest points. calcCombatLevel(skills): RS3 formula including Necromancy. generateSuggestedGoals(players): analyses group averages and returns recommended goals.
utils/auth.jshashPassword(secret): SHA-256 hash. checkGroupAuth(req, res, groupId): middleware that reads X-Group-Password header and validates against DB hash. Returns 401 if wrong, 200 if unclaimed (open access).

Auth model

  • Unclaimed group: anyone can read and write (no password)
  • Claimed group: anyone can read; write operations require the secret in the X-Group-Password header
  • Secret storage: SHA-256 hash in groups.password_hash; plain text is shown once and never stored
  • Per-browser state: which groups you've added = localStorage.myGroupIds; your RSN per group = localStorage.myRsn_{groupId}; unlock state = sessionStorage

12. Suggested Next Steps

🚀 Performance & Reliability

  1. Automated database backups — add a daily cron on the server: cp gim.db ~/backups/gim_$(date +%Y%m%d).db. Consider uploading to Hetzner object storage or a cloud bucket.
  2. Rate limiting — add express-rate-limit middleware to the /api/players/:id/sync endpoint. Jagex's hiscores API can return errors if hammered (e.g. 5 syncs / 30 seconds per group).
  3. Response caching — cache hiscores responses in memory for 5 minutes. Prevents re-fetching when multiple group members sync at the same time.
  4. Database indexes — add CREATE INDEX IF NOT EXISTS idx_skills_player ON skills(player_id) and similar for snapshots. Helps as the group grows. Run EXPLAIN QUERY PLAN <your query> to check.
  5. PM2 cluster mode — if you ever see performance issues under load, PM2 can run multiple Node.js instances. Not needed now but easy to enable.
  6. Server monitoring — set up UptimeRobot (free) to ping https://groupiron.com/api/health every 5 minutes and email you if it goes down.

✨ Features

  1. Discord webhook notifications — when Sync All detects a new 99, big drop, or quest completion in the activity feed, fire a webhook to your group Discord channel. Needs a webhook URL stored as an env variable.
  2. Boss kill counters — hiscores CSV has boss kill count rows (Corporeal Beast, Nex, etc.) — pull and display these in the Leaderboards tab alongside clue scrolls.
  3. Mobile responsive improvements — the skills matrix and leaderboard cards get cramped on mobile. Add media queries for screens under 600px.
  4. Persistent milestone storage — currently milestones are scanned from the last 100 RuneMetrics activities on every page load. If a milestone scrolls out of the feed it's lost. Add a milestones DB table and persist them during sync.
  5. Group progress timeline — a chronological view showing the group's history: who joined, first syncs, milestone dates. Like a "group story" feed.
  6. Diary tracking — RS3 Achievement Diaries (Easy/Medium/Hard/Elite per region). Show which ones players have completed.
  7. Light/dark theme toggle — the UI is dark-only. Add a toggle in the header.
  8. Export data — a button to download the group's data as CSV or JSON for offline reference.
  9. Group notes / shared pinboard — a simple text area per group for shared session notes, boss strategies, loot split rules etc.
  10. Scheduled sync — instead of manual Sync All, let the midnight cron also update RuneMetrics activities so milestones stay fresh automatically.

🔒 Security

  1. CORS restriction — currently CORS is wide open (app.use(cors())). Restrict to https://groupiron.com in production.
  2. Environment variables — move the JWT/secret salt and any future API keys into a .env file using dotenv, not hardcoded. Already gitignored.
  3. Input validation — add express-validator to the group/player creation routes to reject malformed RSNs or oversized inputs.
