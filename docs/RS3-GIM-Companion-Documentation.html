3. SSL & HTTPS

SSL is handled by a Cloudflare Origin Certificate — a certificate issued by Cloudflare to secure the connection between Cloudflare's edge and the origin server (Hetzner). It is not a public CA certificate.

PropertyDetail
Certificate typeCloudflare Origin Certificate (15 years)
Certificate file on server/etc/ssl/cloudflare/origin.pem
Private key file on server/etc/ssl/cloudflare/origin.key
Local backup locationC:\Users\Thomas\OneDrive\RS3 GIM Companion\
Cloudflare SSL modeFull (Strict) — encrypts end-to-end, validates origin cert
Port 80 behaviourNginx redirects all HTTP → HTTPS (301)

Nginx SSL block (summary)

server {
    listen 443 ssl;
    server_name groupiron.com www.groupiron.com;

    ssl_certificate     /etc/ssl/cloudflare/origin.pem;
    ssl_certificate_key /etc/ssl/cloudflare/origin.key;

    # Serve React build
    root /root/RS3-GIM-Companion/client/dist;
    index index.html;
    try_files $uri $uri/ /index.html;

    # Proxy /api to Node.js
    location /api/ {
        proxy_pass http://localhost:3001;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
    }
}

server {
    listen 80;
    server_name groupiron.com www.groupiron.com;
    return 301 https://$host$request_uri;
}
If the cert ever needs renewing: Generate a new Origin Certificate in the Cloudflare dashboard → SSL/TLS → Origin Server → Create Certificate. Save both files and copy them to /etc/ssl/cloudflare/ on the server, then sudo systemctl reload nginx.

6. Deploying Changes

The full workflow is: edit locally → commit → push to GitHub → SSH into server → run deploy script.

Step-by-step

  1. Make your changes in VS Code or with Claude in the Claude desktop app
  2. Claude commits and pushes to GitHub (or do it yourself: git add . && git commit -m "message" && git push)
  3. SSH into the server: ssh root@178.104.126.158
  4. Run the deploy script: bash /root/deploy.sh
  5. The script pulls changes, builds the React frontend, and restarts PM2

What deploy.sh does

cd /root/RS3-GIM-Companion
git pull origin main
npm install --prefix server     # installs any new server packages
npm run build --prefix client   # Vite build → client/dist/
pm2 restart app

Checking the deploy worked

pm2 status          # should show "online"
pm2 logs app --lines 20   # check for startup errors
curl localhost:3001/api/health   # should return {"status":"ok",...}
Typical deploy time: 30–60 seconds. The site is briefly unavailable during PM2 restart (<2 seconds).

8. Requesting Features with Claude

The project is developed using Claude Code (Anthropic's AI coding assistant) inside the Claude desktop app. Claude has access to all project files and can read, edit, and commit code.

How to ask for a feature

  1. Open the RS3 GIM Companion project in Claude desktop app (the working directory is set automatically)
  2. Describe what you want in plain English. Be specific about: where it should appear, what it should show, what data it uses
  3. Claude will read the relevant files, make the changes, and commit + push
  4. You then SSH into the server and run bash /root/deploy.sh
After every feature session: ask Claude "is this pushed to git?" before deploying. Claude will confirm and push if not already done.

Tips for good feature requests

  • Reference existing things: "Like the Skills tab but for bosses" is clearer than "add a boss tab"
  • Describe the data source: "Detect from the RuneMetrics activity feed" or "from hiscores CSV"
  • Describe the layout: "A table with players as columns" or "a grid of cards"
  • One session = one topic: Mixing unrelated tasks in one message works fine, but very large changes across many files are easier to review in separate sessions
  • After deploy, test it: Report back if something looks wrong and Claude will fix it
  • Include screenshots when reporting bugs — they help Claude understand exactly what's wrong

Example prompts that worked well

  • "Add a Leaderboards tab with a Firsts section showing ~40 predefined milestones as cards. Auto-detect from activity feed. Show who achieved each first, or 'not yet' if nobody has."
  • "Add an Achievement Diaries tab. Show 13 regions × 3 tiers. Auto-detect from RuneMetrics activity feed. Allow manual toggle. Two views: grid (regions × tiers table) and per-player."
  • "The boss kill leaderboard should have bosses as rows and players as columns, like the skills matrix. Add a boss filter dropdown. Make Boss Kills the default section."
  • "Do mobile responsive improvements — the skills matrix and leaderboard cards get cramped on mobile. Add media queries for screens under 600px."

9. Database Reference & SQL

Connect to the database

Useful SQL queries

See all groups and their claim status

SELECT id, name, gim_type, (password_hash IS NOT NULL) as claimed, last_activity
FROM groups;

Reset a group to unclaimed (if secret is lost)

-- You can't recover the secret (it's hashed). To reset so group becomes unclaimed:
UPDATE groups SET password_hash = NULL WHERE name = 'True Deciples';

See all players in a group

SELECT p.rsn, p.combat_level, p.last_synced, s.level as total_level, s.xp as total_xp
FROM players p
LEFT JOIN skills s ON s.player_id = p.id AND s.skill_name = 'Overall'
WHERE p.group_id = 1
ORDER BY s.xp DESC;

Check a player's clue scroll counts

-- Clue scrolls are stored in stats_json from hiscores (rows 55-59 in the CSV)
SELECT rsn, json_extract(stats_json, '$.activities') FROM players WHERE rsn = 'YourRSN';

Check boss kill counts for a group

SELECT p.rsn, bk.boss_key, bk.kills, bk.last_seen
FROM boss_kills bk
JOIN players p ON p.id = bk.player_id
WHERE p.group_id = 1 AND bk.kills > 0
ORDER BY bk.boss_key, bk.kills DESC;

Check achievement diary completions for a group

SELECT p.rsn, a.key, a.achieved, a.achieved_at, a.manual
FROM achievements a
JOIN players p ON p.id = a.player_id
WHERE p.group_id = 1 AND a.achieved = 1
ORDER BY p.rsn, a.key;

Reset boss kills for a player (to re-accumulate from scratch)

DELETE FROM boss_kills WHERE player_id = (SELECT id FROM players WHERE rsn = 'YourRSN');
DELETE FROM boss_kill_log WHERE player_id = (SELECT id FROM players WHERE rsn = 'YourRSN');

List all drops for a group

SELECT p.rsn, d.item_name, d.notes, d.dropped_at
FROM drops d
JOIN players p ON p.id = d.player_id
WHERE p.group_id = 1
ORDER BY d.dropped_at DESC;

Count goals by status

SELECT status, COUNT(*) FROM goals GROUP BY status;

Show XP snapshots for a player (for debugging weekly XP)

SELECT snapshot_date, total_xp
FROM snapshots
WHERE player_id = (SELECT id FROM players WHERE rsn = 'YourRSN')
ORDER BY snapshot_date DESC
LIMIT 10;

Wipe everything and start fresh

DELETE FROM achievements; DELETE FROM boss_kills; DELETE FROM boss_kill_log;
DELETE FROM drops; DELETE FROM item_requests; DELETE FROM goals;
DELETE FROM goal_contributors; DELETE FROM snapshots;
DELETE FROM skills; DELETE FROM players; DELETE FROM groups;
DELETE FROM sqlite_sequence;

Boss kill detection — how it works

RS3 does not have a public boss kill hiscores API (unlike OSRS). Kills are accumulated from the RuneMetrics activity feed:

  1. On each sync, autoCountBossKills(playerId, activities) processes the last 100 RuneMetrics activities
  2. Each activity is checked against 26 boss patterns (most specific first, e.g. "Nex: Angel of Death" before "Nex")
  3. Activities matching SKIP_PATTERNS (drops, level-ups, diary completions, quest completions) are ignored
  4. If the activity details field contains an absolute kill count (e.g. "Kill count: 157"), the boss_kills row is upserted with MAX(existing, new_count)
  5. Otherwise, a unique signature for the activity is inserted into boss_kill_log (INSERT OR IGNORE). If the row is new (not seen before), kills increments by 1
  6. Kill counts grow over time. They start at 0 for any boss not yet seen in the feed

Auth model

  • Unclaimed group: anyone can read and write (no password)
  • Claimed group: anyone can read; write operations require the secret in the X-Group-Password header
  • Secret storage: SHA-256 hash in groups.password_hash; plain text is shown once and never stored
  • Per-browser state: sidebar groups = localStorage.myGroupIds; favourite groups = localStorage.favoriteGroupIds; RSN per group = localStorage.myRsn_{groupId}; unlock state = sessionStorage

12. Suggested Next Steps

🚀 Performance & Reliability

  1. Automated database backups — add a daily cron on the server: cp gim.db ~/backups/gim_$(date +%Y%m%d).db. Consider uploading to Hetzner object storage.
  2. Rate limiting — add express-rate-limit middleware to the sync endpoints. Jagex's APIs can return errors if hammered.
  3. Response caching — cache hiscores responses in memory for 5 minutes. Prevents re-fetching when multiple members sync at the same time.
  4. Database indexes — add indexes on boss_kills(player_id), achievements(player_id), skills(player_id). Run EXPLAIN QUERY PLAN to verify.
  5. Server monitoring — set up UptimeRobot (free) to ping https://groupiron.com/api/health every 5 minutes and email you if it goes down.

✨ Features

  1. Discord webhook notifications — when Sync All detects a new 99, big drop, quest completion, or diary tier in the activity feed, fire a webhook to your group Discord channel.
  2. Persistent milestone storage — currently milestones are scanned from the last 100 RuneMetrics activities on every page load. If a milestone scrolls out of the feed it's lost. Add a milestones DB table and persist them during sync.
  3. Group progress timeline — a chronological view showing the group's history: who joined, first syncs, milestone dates. Like a "group story" feed.
  4. Light/dark theme toggle — the UI is dark-only. Add a toggle in the header.
  5. Export data — a button to download the group's data as CSV or JSON for offline reference.
  6. Group notes / shared pinboard — a simple text area per group for shared session notes, boss strategies, loot split rules etc.
  7. Elite Diary tier — currently tracking Easy/Medium/Hard. Add Elite tier to the Achievement Diaries tab (if detectable in the RuneMetrics feed).
  8. Boss kill history chart — a line chart per boss showing kill count growth over time (would need a boss_kill_snapshots table similar to XP snapshots).

🔒 Security

  1. CORS restriction — currently CORS is wide open (app.use(cors())). Restrict to https://groupiron.com in production.
  2. Environment variables — move any future API keys into a .env file using dotenv. Already gitignored.
  3. Input validation — add express-validator to group/player creation routes to reject malformed RSNs or oversized inputs.
